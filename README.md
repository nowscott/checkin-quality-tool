# 打卡质检数据生成工具

纯前端课堂反馈打卡质检工具。用户上传课堂反馈质检名单和企微聊天导出数据后，系统在浏览器本地完成 Excel 读取、数据清洗、教师与学员匹配，并生成可追溯的多 Sheet Excel。

- 生产地址：[https://daka.0211120.xyz](https://daka.0211120.xyz)
- 技术栈：Vite 8、React 19、TypeScript 6、Web Worker
- Excel 读取：SheetJS 0.20.3 浏览器构建
- Excel 导出：fflate 流式生成 OOXML/XLSX
- 当前版本：`v2.3.1`

项目没有文件上传接口、后端服务、数据库或分析脚本。教师、学员和聊天数据只在当前浏览器中处理。

## 业务目标

课堂反馈名单中，每位教师的每位学员只保留一个质检课次。系统再去聊天导出数据中确认：该教师是否在对应学员群内发送过包含学员姓名关键词的打卡内容。

判断公式：

```text
同一教师邮箱
+
（群名包含学员关键词 或 聊天内容包含学员关键词）
=
已发送
```

教师身份只通过邮箱确认。企微昵称、聊天发送人名称和群名中的教师名称都不用于确认教师身份。

## 输入文件

### 课堂反馈质检名单

系统会寻找同时包含以下表头的工作表：

```text
老师姓名、学员姓名、老师邮箱、课次开始时
```

当前还会读取：

```text
学员号、间（第一列作为开始时间、第二列作为结束时间）、
校区、项目组、科目
```

### 聊天质检明细

系统会寻找同时包含以下表头的工作表：

```text
聊天类型、发送方、聊天内容
```

当前还会读取：

```text
姓名、邮箱、群名/好友昵称、群聊发送人名称、
群聊发送人邮箱、聊天时间
```

只支持 `.xlsx`。不要直接加入 `.xls` 支持，除非已经对大文件、日期和中文字段完成单独验证。

## 完整处理流程

### 1. 页面读取配置

React 页面负责：

1. 接收两个 Excel 文件。
2. 读取 `public/data/whitelist.csv`。
3. 接收服务周和末字弱匹配设置。
4. 启动 TypeScript Web Worker。
5. 展示处理进度并下载结果。

文件通过结构化克隆传给 Worker，不会发送到服务器。

### 2. 解析 Excel

Worker 使用 `public/vendor/xlsx.full.min.js` 中的 SheetJS 0.20.3，并启用：

```ts
{
  type: "array",
  dense: true,
  cellDates: true
}
```

SheetJS 当前保留为经过验证的浏览器构建，不使用 npm 上版本较旧的 `xlsx` 包替换。

### 3. 名单预处理

名单处理规则：

1. 找到目标工作表和字段位置。
2. 清洗教师邮箱并统一转为小写。
3. 清洗学员姓名。
4. 按“教师邮箱 + 清洗后学员姓名”分组。
5. 每组按上课日期、上课开始时间、源行号排序。
6. 保留最早的一条课次。
7. 将该组原始数量记录为“该周课次数”。

教师邮箱为空时，分组键会临时回退为教师姓名，但后续无法匹配到聊天邮箱，通常会得到“未发送”。

### 4. 学员姓名清洗

姓名会先去除空格，然后循环清理以下末尾标记：

```text
括号备注
半角或全角数字
“重复”
“学员 / 同学 / 学生”
“家长 / 妈妈 / 爸爸 / 姐姐 / 哥哥 / 妹妹 / 弟弟”
“初一至初三 / 高一至高三”及相应数字形式
```

特殊规则：

- 只有“四字中文姓名且末尾为大写中文一”时，才删除最后的“一”。
- 三字姓名如“宋卓一”不会删除“一”。
- “陈同学”会清洗为“陈”，自动进入单字弱匹配。
- 原始姓名、清洗后姓名和清洗说明都会写入结果，便于追溯。

### 5. 服务周识别

页面允许选择：

```text
自动识别、第一周、第二周、第三周、第四周、第五周
```

自动识别时：

1. 根据每条课次日期计算当月第几周，周一作为一周开始。
2. 统计各服务周的课次数量。
3. 选择数量最多的服务周。
4. 将识别方式和各周分布写入“处理说明”。

文件名中的日期只用于页面提示，最终自动结果以名单课次日期为准。

### 6. 聊天预处理

以下聊天会被删除：

- `聊天类型`等于“私聊”。
- `群聊发送人邮箱`和回退字段`邮箱`都没有有效邮箱。
- `发送方`存在且不等于“员工”。
- 被当前正则判断为引用回复。

有效教师邮箱取值顺序：

1. `群聊发送人邮箱`
2. `邮箱`

清洗后聊天只保留：

```text
有效教师邮箱、邮箱来源、发送人名称、群名/好友昵称、
聊天时间、聊天内容、源聊天行号
```

### 7. 引用回复识别

当前逻辑要求一条消息同时满足：

- 开头类似 `「某人：` 或 `『某人：`
- 内容中存在较长的连续分隔线

满足后会删除整条聊天，避免被引用的旧课堂反馈重复参与匹配。

注意：当前不是“只截掉引用部分”，而是删除整条消息，这是一个已知待改进点。

### 8. 白名单

白名单文件：

```text
public/data/whitelist.csv
```

CSV 字段：

```text
学员号,学员姓名,处理方式,匹配别名,说明
```

关联优先级：

1. 学员号
2. 清洗后的学员姓名

处理方式：

- `免检`：不检查聊天，`发送情况`直接写“已发送”，`匹配结论`写“白名单免检”。
- `别名`：将别名后两字加入匹配候选，但仍须真实聊天命中后才算已发送。

多个别名使用 `|`、`;` 或 `；` 分隔。

白名单不需要教师姓名和教师邮箱。需要新增特殊学员时，优先填写学员号，避免同名学员互相影响。

### 9. 教师匹配

教师匹配只比较邮箱，英文大小写忽略。

不使用以下字段确认教师身份：

- 质检名单教师姓名
- 企微昵称
- 群聊发送人名称
- 群名中的教师名称

因此教师姓名带数字、企微昵称与本名不一致，都不会影响匹配。

### 10. 学员关键词匹配

在同一教师邮箱下，检查群名和聊天内容，两者取并集。

强匹配：

```text
清洗后学员姓名至少两字
→ 使用姓名后两字
→ 群名或聊天内容任一处包含关键词
```

示例：

```text
侯美玲 → 关键词“美玲”
群名“美玲家长群” → 已发送
聊天内容“美玲本周完成情况” → 已发送
```

弱匹配：

- 清洗后姓名不足两字时，自动使用末字匹配。
- 页面开启“末字弱匹配”后，普通姓名在后两字未命中时也尝试末字。
- 弱匹配仍要求教师邮箱一致。

英文姓名、英文群名和英文聊天内容统一转为小写后比较。

别名匹配：

- 白名单别名作为实际学员姓名证据，优先于登记姓名匹配。
- 使用白名单别名的后两字。
- 必须真实命中群名或聊天内容。

### 11. 最佳匹配与结果

同一学员可能命中多条聊天。当前排序规则：

1. 白名单别名匹配优先。
2. 强匹配其次。
3. 弱匹配最后。
4. 同级时按聊天时间从早到晚。

第一条作为“打卡结果”的主要命中证据，所有命中记录都写入“匹配明细”。

### 12. 教师姓名输出

“打卡结果”第一列教师姓名以上传的质检名单为准。

若名单教师邮箱 `@` 前的本地部分以数字结尾，则：

1. 删除名单教师姓名原有的半角或全角数字后缀。
2. 追加邮箱本地部分末尾数字。

例如：

```text
名单教师姓名：吴君怡
名单教师邮箱：wujunyi7@example.com
输出教师姓名：吴君怡7
```

聊天发送人昵称不参与教师姓名生成。

## 输出 Excel

### Sheet 1：打卡结果

每位教师、每位学员一行，包含最终发送情况、匹配结论和主要证据。

核心前六列：

```text
教师姓名、上课日期、上课时间、学生姓名、服务周、发送情况
```

后续列包含序号、教师邮箱、清洗后姓名、命中关键词、命中位置、群名、白名单说明、源行号等追溯信息。

### Sheet 2：匹配明细

记录每一条实际命中的聊天，包括：

```text
教师邮箱、原始学员姓名、清洗后姓名、关键词、匹配强度、
命中位置、群名、聊天时间、聊天内容、源聊天行号
```

### Sheet 3：清洗后聊天

只包含参与匹配的有效聊天及必要字段。

### Sheet 4：处理说明

记录：

- 源文件名和目标工作表。
- 服务周及识别方式。
- 所有主要清洗和匹配规则。
- 名单、聊天和匹配阶段的数量统计。
- 生成时间。

### Sheet 5：内置白名单

记录本次运行实际使用的白名单条目，不包含教师信息。

## 技术架构

```text
React 页面
  ├─ 上传文件、选项和状态展示
  ├─ 读取内置白名单 CSV
  └─ 启动 Vite 打包的经典 Web Worker
       ├─ SheetJS 解析名单与聊天
       ├─ 名单清洗和去重
       ├─ 聊天清洗
       ├─ 白名单关联
       ├─ 邮箱与姓名关键词匹配
       └─ fflate 流式生成多 Sheet XLSX
```

Worker 使用经典 IIFE 格式，以便继续通过 `importScripts` 加载现有 SheetJS 浏览器构建：

```ts
worker: {
  format: "iife"
}
```

不要未经回归验证将 Worker 改为 module Worker，也不要直接替换 SheetJS 版本。

- `src/worker/progress.ts`：处理进度消息。

- `public/vendor/xlsx.full.min.js`：SheetJS 0.20.3 浏览器构建。

## 已知问题与待解决事项

### P1：单字弱匹配误判风险

异常姓名清洗后只剩一个字时会自动弱匹配。例如“陈同学”清洗为“陈”，同一教师邮箱下群名或内容只要出现“陈”就可能判定已发送。

建议方向：

- 单字匹配增加群名优先或固定格式限制。
- 将单字结果改为“待人工复核”，而非直接等同正常已发送。
- 结合学员号维护更明确的别名或群名白名单。

### P1：聊天没有按服务周严格限定

当前依赖用户上传正确日期范围的聊天导出文件，匹配时没有再次校验聊天时间是否属于目标服务周。聊天文件范围过宽时，历史消息可能命中。

建议方向：

- 从名单日期推导明确的起止日期。
- 解析聊天时间并在匹配前过滤。
- 将实际过滤日期写入“处理说明”。

### P1：引用回复会删除整条消息

当前识别到引用回复后会删除整条聊天。如果分隔线后的新增回复本身包含有效学员姓名，也会一起丢失，可能造成漏判。

建议方向：

- 将引用内容和新增内容拆分。
- 只删除引用区，保留分隔线后的新消息。
- 为多种企微引用格式增加样本测试。

### P2：缺少学员号时可能误用同名白名单

白名单按学员号优先，缺少学员号时全局按清洗后姓名关联。同名学员可能错误继承免检或别名规则。

建议方向：

- 免检条目强制要求学员号。
- 姓名回退只允许别名，不允许免检。
- 导出白名单审计告警，列出按姓名回退的记录。

### P2：自动服务周在跨月数据中可能不符合业务周

当前服务周是“月内第几周”，并选择课次数量最多的周。跨月、跨周或日期分布异常时，可能需要人工选择。

建议方向：

- 明确业务上的服务周定义。
- 使用实际日期区间而不是月内序号作为内部依据。
- 自动识别存在多个主要周时提示用户确认。

### P2：源表结构变化会导致失败

当前依赖固定中文表头及两个名为“间”的重复字段。上游改名、增加空格、调整重复列顺序后，可能找不到字段或读错时间列。

建议方向：

- 建立字段别名表。
- 在处理前展示识别到的字段映射。
- 对缺失字段给出具体工作表和表头诊断。

### P2：大文件受浏览器内存限制

已验证约 8.5 MB 名单和 54 MB 聊天文件，但 SheetJS 仍会在内存中解析完整工作簿，随后还会保留清洗数据、匹配结果和导出缓冲区。数百 MB 文件可能导致标签页崩溃。

建议方向：

- 上传后先提示文件大小和内存风险。
- 调研支持真正流式读取 XLSX 的方案。
- 限制导出“匹配明细”中过量重复消息。
- 大规模场景另建本地桌面版或服务端批处理版。

### P3：缺少系统化自动测试

目前有 TypeScript 检查、生产构建和真实大文件回归，但没有完整的单元测试和端到端测试。

建议优先补充：

- 姓名清洗测试。
- 教师邮箱大小写测试。
- 单字、后两字、英文和别名匹配测试。
- 免检与同名学员测试。
- 引用回复拆分测试。
- 跨月服务周测试。
- 固定小型 XLSX 输入输出快照测试。

### P3：SheetJS 类型声明是最小手写声明

`src/worker/sheetjs.d.ts` 只声明了项目当前使用的 API。后续新增 SheetJS 功能时，TypeScript 不会自动获得完整官方类型。

建议方向：

- 继续保持最小声明并随使用范围补充。
- 若更换 SheetJS 来源或版本，先验证 API、日期解析和 dense 模式行为。

## 数据安全

- 不要将真实源 Excel、生成结果、教师邮箱、学员姓名或聊天内容提交到 Git。
- 回归文件使用本机绝对路径传给脚本。
- Vercel 只托管静态网页和项目内置白名单。
